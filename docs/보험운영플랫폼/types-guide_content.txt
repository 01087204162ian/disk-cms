티켓 유형 가이드

기본 원칙

- 모든 업무는 아래 9가지 티켓 유형 중 하나로만 시작합니다 (복수 성격일 경우, "시작 목적" 기준으로 선택)
- 티켓 유형은 일을 제한하려는 게 아니라, 일을 보호하기 위한 분류입니다.

9가지 티켓 유형

1. 리서치 (RESEARCH)
신시장, 신규 고객, 신규 제휴 가능성을 탐색하는 모든 활동

포함 업무:
- 신규 시장 조사
- 잠재 고객 리스트 발굴
- 보험사/플랫폼 후보 조사
- 레퍼런스 수집

산출물:
- 고객/파트너 리스트
- 시장 분석 메모
- 접촉 이력

특징:
❌ 계약 없음
❌ 금액 없음
✅ 출발점

2. 프로젝트 (PROJECT)
보험 신상품 개발 또는 특정 목표를 가진 고객 접촉 활동

포함 업무:
- 보험 신상품 기획 프로젝트
- 특정 기업/플랫폼 대상 영업
- 제휴 추진 TF
- PoC, 시범 운영

산출물:
- 프로젝트 계획
- 진행 이력
- 결과 요약

특징:
✅ 티켓 파생
✅ 기간 중심

3. 계약 (CONTRACT)
보험 계약의 체결 또는 갱신과 직접적으로 관련된 업무

포함 업무:
- 신규 보험 계약
- 계약 갱신
- 조건 변경
- 계약 종료 처리

산출물:
- 계약서
- 증권번호
- 계약 조건 기록

특징:
✅ 고객/보험사 연결
✅ 기준점

4. 정산 (SETTLEMENT)
보험 계약에 따른 금액 정산 업무

포함 업무:
- 일/주/월/연 단위 정산
- 보험사 정산 자료 검증
- 입금 확인
- 차이 발생 시 조정

산출물:
- 정산 결과
- 정산 기준 기록
- 정산 이슈 로그

특징:
✅ 반복성
❌ 리스크 큼
✅ 체크리스트 필수

5. 실적 (PERFORMANCE)
계약 결과를 보험사 시스템 실적과 대조·검증하는 업무

포함 업무:
- 보험사 시스템 실적 대조
- 수수료 확인
- 수수료 배분
- 실적 차이 분석

산출물:
- 실적 비교 결과
- 수수료 산출 근거
- 배분 내역

특징:
✅ 정산 이후
✅ 이력 중요

6. 클레임 (CLAIM)
보험 계약 이후 발생하는 모든 사후 처리 (After Service)

포함 업무:
- 사고 접수
- 고객 민원
- 보험사 클레임 대응
- 분쟁/이슈 처리

산출물:
- 사고/민원 처리 이력
- 보험사 커뮤니케이션 기록
- 결과 보고

특징:
❌ 리스크
❌ 민감도 높음
✅ 기록 필수

7. 개발 (DEV)
시스템, API, 앱/웹 개발과 관련된 모든 기술 업무

포함 업무:
- API 개발
- CMS 기능 개발
- 앱/웹 화면 개발
- 시스템 연동
- 버그 수정

산출물:
- 코드
- API 명세
- 배포 이력

특징:
✅ 티켓 파생
✅ 기술 로그

8. 기획 (PLAN)
프로젝트 확정 이후, 실제 실행을 위한 상세 설계 업무

포함 업무:
- 업무 프로세스 설계
- 시스템 연동 기획
- 운영 정책 정의
- 화면/UX 기획

산출물:
- 기획서
- 정책 문서
- 연동 정의서

특징:
✅ 프로젝트→기획→개발
✅ 기록 핵심

9. 기타 (OTHER)
위 유형에 명확히 속하지 않는 예외 업무

사용 원칙:
- 가급적 사용 ❌
- 임시 성격 업무
- 이후 유형 재분류 권장

특징:
❌ 최소화

티켓 유형 간 자연스러운 흐름

리서치 (시장/고객 탐색)
↓
프로젝트 (목표 기반 추진)
↓
기획 (상세 설계)
→
개발 (시스템 구현)
↓
계약 (보험 체결)
↓
정산 (금액 정산)
→
실적 (결과 검증)
↓
클레임 (사후 처리)

티켓 유형은 고립된 분류가 아니라 흐름입니다.

빠른 판단 가이드 (직원용)

"이 일을 왜 시작했는가?"

시작 목적 → 티켓 유형
새로운 가능성 탐색 → 리서치
특정 목표 달성 → 프로젝트
보험 계약/갱신 → 계약
금액 계산/정산 → 정산
실적 검증/대조 → 실적
사고/민원 처리 → 클레임
시스템 개발/수정 → 개발
업무/시스템 설계 → 기획
위에 해당 없음 → 기타 (최소화)

실무 적용 예시

예시 1: 신규 보험상품 개발
1. 리서치 티켓: 시장 조사, 경쟁사 분석
2. 프로젝트 티켓: "홀인원 보험 개발 프로젝트"
3. 기획 티켓: 상품 구조, 약관, 프로세스 설계
4. 개발 티켓: 시스템 개발, API 연동
5. 계약 티켓: 첫 계약 체결
6. 정산 티켓: 월별 정산
7. 실적 티켓: 보험사 실적 대조

예시 2: 일상 업무
• 매일 오전 9시 발송 → 정산 티켓 (반복 업무)
• 고객 사고 접수 → 클레임 티켓
• 계약 갱신 요청 → 계약 티켓
• 시스템 버그 발견 → 개발 티켓

주의사항

유형 선택 시 헷갈리는 경우

Q: 프로젝트와 기획의 차이는?
- 프로젝트 = 큰 목표 단위 (예: "신상품 개발")
- 기획 = 프로젝트 내 상세 설계 단계

Q: 정산과 실적의 차이는?
- 정산 = 우리가 계산한 금액
- 실적 = 보험사 시스템과 대조·검증

Q: 여러 유형에 걸쳐 있으면?
- "시작 목적" 기준으로 선택
- 이후 다른 티켓으로 파생 가능

체크리스트 (티켓 생성 전)

☐ 담당자(Owner)가 명확한가?

☐ 완료 기준이 명확한가?

핵심 메시지

유형이 명확할수록, 책임·기준·승인이 쉬워집니다

티켓 유형은 여러분의 일을 분류하는 게 아니라,
여러분의 일을 보호하기 위한 구조입니다.

유형 분류가 어렵거나 애매한 경우, 팀장 또는 관리자에게 문의하세요.
